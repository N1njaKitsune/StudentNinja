This document specifies every change required to take the student profile from its current state to a production-ready screen, covering two parallel workstreams: completing the SVG avatar character to member-ready quality, and rebuilding the aura system on canvas so it conforms to the body silhouette with organic noise and cinematic transitions.

The two workstreams below are independent — the avatar SVG upgrade can be completed and tested first, before the canvas engine is written. Build in the order listed at the end of this document.

Part 2
Avatar SVG — Member-Ready Spec

The goal is a stylised martial arts character that reads clearly as a solid form in the dark environment, while remaining appropriate for the aura to illuminate from outside. The aesthetic should feel like the character exists in low light — dark fabric, warm skin, the belt as the one vivid accent — with the aura glow providing the surrounding luminosity. Not photorealistic. Stylised and purposeful.

The key principle: the aura lights the character, not the character itself. This means fills stay subdued so that when the aura color falls across the figure it reads as genuine illumination rather than competing with a brightly-lit avatar.

Part 3
Canvas Aura System — Architecture

The CSS aura ring is replaced entirely by a canvas element. The key advance over the prototype aura (v1.3) is that instead of uniform elliptical gradients, the aura is rendered as a dilated, blurred version of the actual character silhouette. The body shape defines the glow shape.

Why canvas over CSS
CapabilityCSS (current)Canvas (target)
Aura conforms to body shapeNo — generic ovalYes — path dilation
Organic noise / edge irregularityNoYes — sine perturbation per frame
Motes launch from body surfaceNo — fixed ovalYes — points sampled from silhouette path
Colour transitionsCSS transition (OK)Smooth per-frame interpolation
Rim lightingApproximated with box-shadowTrue edge stroke on silhouette path
Performance on mobileVery goodGood (2D canvas, no WebGL needed)
Core technique — silhouette dilation

The silhouette of the character is defined as a polygon/bezier path in canvas coordinates. Each aura "layer" is produced by:

// Pseudocode — per frame, per layer function renderAuraLayer(ctx, silhouettePath, layerIndex, density, time) { const outset = LAYER_OUTSET[layerIndex]; // px outset from silhouette const blur = LAYER_BLUR[layerIndex]; // gaussian blur radius const maxOpacity = LAYER_MAX_OP[layerIndex]; // 1. Perturb the silhouette points with organic noise const noisyPath = perturbPath(silhouettePath, time, layerIndex, density); // 2. Expand (dilate) the path outward by `outset` px const dilatedPath = expandPath(noisyPath, outset); // 3. Draw on offscreen canvas, fill in aura colour offCtx.clearRect(0, 0, w, h); offCtx.fillStyle = auraColor(currentStage, layerOpacity); offCtx.fill(dilatedPath); // 4. Apply blur filter offCtx.filter = `blur(${blur}px)`; // 5. Composite onto main canvas ctx.globalAlpha = layerProgress * maxOpacity * surgeBoost; ctx.drawImage(offscreenCanvas, 0, 0); }
Silhouette path — canvas coordinates

The SVG uses viewBox 0 0 130 285. The char-wrap on iPhone is 124×295px, giving scale factors of 0.954x and 1.035y. The canvas is positioned 40px outside the char-wrap on all sides (so canvas = 204×375px on iPhone, 280×520px on iPad), offset by left:-40px top:-40px. The silhouette path below is in canvas-local coords with the 40px offset already added.

// 22-point outline path — canvas coords (iPhone, canvas=204×375) // Drawn as quadratic beziers between anchor points // Counter-clockwise from hair apex const SILHOUETTE_POINTS = [ // Head & hair { x: 99, y: 63 }, // hair apex { x: 73, y: 67 }, // hair top-left { x: 72, y: 94 }, // head left // Left side — shoulder, arm, hand { x: 65, y: 158 }, // left shoulder { x: 63, y: 192 }, // left arm outer mid { x: 64, y: 215 }, // left hand tip { x: 71, y: 224 }, // left wrist re-entry // Left leg & foot { x: 73, y: 256 }, // left hip { x: 71, y: 300 }, // left knee outer { x: 72, y: 329 }, // left ankle { x: 70, y: 340 }, // left heel { x: 84, y: 342 }, // left foot far end // Right foot & leg { x: 118, y: 341 }, // right foot far end { x: 126, y: 336 }, // right heel { x: 124, y: 325 }, // right ankle { x: 120, y: 295 }, // right knee outer { x: 120, y: 256 }, // right hip // Right arm & shoulder { x: 127, y: 222 }, // right wrist re-entry { x: 134, y: 213 }, // right hand tip { x: 135, y: 191 }, // right arm outer mid { x: 133, y: 157 }, // right shoulder { x: 126, y: 93 }, // head right { x: 125, y: 67 }, // hair top-right // close to apex ]; // iPad: multiply all coords by 1.28 (160/124) // Scale factor stored as SILHOUETTE_SCALE_IPAD = 1.28

Noise / organic edge parameters
// Each outline point is perturbed each frame using overlapping sine waves // The perturbation is small — aura feels alive, not distorted function perturbPoint(point, pointIndex, layerIndex, time, density) { const FREQ = [0.40, 0.55, 0.70, 0.85, 1.0]; // Hz per layer const OFFSET = [0.0, 1.3, 2.6, 3.9, 5.2]; // phase per layer const maxAmp = 1.2 + density * 2.0; // 1.2px at min density → 3.2px at max const t = time * FREQ[layerIndex] + OFFSET[layerIndex]; const phi = pointIndex * 0.618; // golden ratio spacing per point return { x: point.x + Math.sin(t + phi) * maxAmp, y: point.y + Math.cos(t * 0.8 + phi) * maxAmp * 0.6, // less vertical noise }; }
Mote system (updated for canvas)

In v1.3 motes spawned on a fixed oval. In v4.0 they spawn at random points on the silhouette path, so they drift upward from the character's actual surface rather than thin air.

// Sample a random point on the silhouette outline function randomSilhouettePoint(path) { const i = Math.floor(Math.random() * path.length); const j = (i + 1) % path.length; const t = Math.random(); return { x: path[i].x + (path[j].x - path[i].x) * t, y: path[i].y + (path[j].y - path[i].y) * t, }; } // Mote count by density (same curve as v1.3) const moteCount = density < 0.40 ? 0 : Math.round((density - 0.40) / 0.60 * 18); // Standard tier: cap at 12. Enhanced tier: cap at 24.
Rim lighting — dark stages (10–13)

For stages 10–13 (Shinku through Stage 13), the outer aura is very dark and the glow would be invisible against the background. The solution is rim lighting: after drawing the aura layers, draw the silhouette path as a coloured stroke, blurred slightly, so it appears as light emanating from the body's edge.

StageRim colour (bright version)Stroke widthBlur
Shinku (9)rgba(220, 90, 90, rimOp)3px6px
Kurenai (10)rgba(200, 80, 140, rimOp)3px7px
Tasogare (11)rgba(170, 100, 210, rimOp)4px8px
Stage 13 (12)rgba(200, 130, 255, rimOp)4px10px

rimOp = 0.18 + density × 0.40 — scales with how much XP the student has earned in that stage.

Colour bleed — 90%+ XP

Same logic as v1.3 but rendered on canvas. When XP ≥ 90, an additional L5-equivalent layer is drawn in the next stage's colour at very low opacity, composited on top of the outer aura layers. Strength ramps from 0 at 90% to max at 100%.

if (xp >= 90 && currentStage < 12) { const bleedStrength = (xp - 90) / 10; // 0 → 1 const bleedOp = bleedStrength * 0.10; // Draw L5 in next stage colour at bleedOp opacity renderAuraLayer(ctx, silhouette, 4, density, time, GLOW_RGB[currentStage + 1], bleedOp); }
Cinematic level-up transition

Identical 4-phase sequence to v1.3, now implemented on canvas rather than CSS:

PhaseTimingCanvas implementation
1. Contract0–300msScale the canvas element to 0.85 using CSS transform. Simultaneously reduce all layer outsets toward 0. The silhouette appears to draw inward.
2. Burst300–600msDraw a radial gradient flash on canvas in white blending to next stage colour. Fade out over 300ms while canvas scales back toward 1.0.
3. Stage change600msUpdate currentStage and reset XP to 0. New colour loads into GLOW_RGB slot. Canvas spring-scales back to 1.0 with overshoot (cubic-bezier 0.34, 1.56, 0.64, 1).
4. Settle600–1400msNormal render loop resumes. New colour fades in over ~800ms as density is at floor (0.25). isTransitioning flag cleared at 1400ms.
Performance tiers
TierFPSLayersMotesNoisePixel ratioWhen applied
ReducedStatic — render once on state changeL1–L2 only0None — static path1×User-selected, or detected low RAM
Standard30fpsL1–L5Max 12±1.5px1.5×Default for all devices
Enhanced60fpsL1–L5 + sub-layers in surgeMax 24±3px2×User-selected
Detection heuristic (auto): On first render, time one full frame. If >33ms (below 30fps), silently drop to Reduced. Otherwise run Standard. User can always override in settings.

Belt colour → aura
Recommendation: keep these two systems completely independent. The belt tracks physical dojo rank. The aura tracks Nendō progression. Mixing them visually would blur two distinct achievement systems. The belt stays as the vivid accent on the character body; the aura is its own surrounding field. No data connection needed.
Aura stage label — left column sync

The left column already shows "Stage 2 · Hikari" in the .id-aura div. This is currently static. In production, this text should be driven by the same stage value passed to AuraController. Wire them to the same data source — don't duplicate.

Part 5
Build Order

Each step is independently testable. Never start step N until step N-1 is visually confirmed.

1
Avatar SVG upgrade — male character
Apply all fill/stroke changes from Part 2 to the existing SVG in StudentProfile_v3.0.html. Save as v3.1. Confirm visually at both iPhone and iPad sizes. The character should read as a solid dark-clad martial artist with a warm face and vivid belt. The aura ring stays as CSS at this stage — only the character changes.
2
Canvas — static proof of concept
Add the canvas element to v3.1. Write the silhouette path, the path dilation function, and render a single static aura in Stage 3 (Kogane, gold) at 60% density. No animation, no motes. Just confirm the shape conforms to the body. This is the technically risky step — prove it before adding complexity.
3
Canvas — breathing animation + noise
Add the requestAnimationFrame loop. Connect the sine-wave noise perturbation. All 5 layers breathing at their staggered frequencies. Density controlled by a test slider. Confirm Standard quality tier performance on a real device (or throttled Chrome DevTools).
4
Connect all 13 stages + density system
Port the STAGES data and GLOW_RGB table from the aura prototype. Wire stage and XP to the canvas renderer. Connect the density formula (floor 0.15 for Tomoshibi, 0.25 for all others). Add the belt switcher from v3.0 so stage changes visually in the profile context.
5
Surge, corona, colour bleed
Add the 80% XP surge mode (faster breathing), the corona ring (now a canvas stroke, not a CSS div), and the 90%+ colour bleed from the next stage. Add the XP bar shimmer. Port directly from v1.3 logic.
6
Motes
Add floating mote particles spawning from the silhouette outline rather than a fixed oval. Use the randomSilhouettePoint() function from Part 3. Confirm mote count scaling with density and the performance cap per quality tier.
7
Rim lighting (dark stages)
Add the rim stroke pass for stages 10–13. Draw the silhouette as a blurred coloured stroke after the aura fill layers. Confirm visibility of Stage 13 (the deep violet) against the dark background.
8
Cinematic level-up
Build the 4-phase transition. Contract, burst, stage change, settle. isTransitioning flag prevents user input during sequence. Run through all 12 transitions (stage 1→2 through 12→13) and confirm each looks correct — the burst colour should match the incoming stage's glow, not white.
9
Performance tiers + quality setting
Wrap the renderer in the quality tier system. Add the auto-detection heuristic (time first frame). Add the user-facing quality toggle (a small settings entry in the profile — not prominent, but accessible). Confirm Reduced tier works as a fully static fallback.
10
Female variant SVG
Build the female character SVG using the modification spec from Part 2. Drop in as an alternate SVG block, toggled by a gender flag. The canvas aura requires no changes — the silhouette path will need a matching female version (same structure, adjusted proportions).
11
React component wrap
Extract the canvas controller and SVG into a self-contained React component AuraAvatar with props: stage, xp, gender, beltColor, beltStripeColor, quality. Matches inline-style architecture of the rest of the app. Cleanup on unmount. This is the production deliverable.
